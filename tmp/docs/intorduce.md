프로젝트마다 중점적으로 고민했던 방향이 달랐기 때문에, 두 개의 프로젝트를 나누어 소개하고자 합니다. 하나는 기술적으로 굉장히 깊은 고민을 했던 프로젝트이고 다른 하나는 사용자 경험을 굉장히 신경을 썼던 프로젝트입니다. 먼저 기술적으로 고민을 가장 깊이 했던 프로젝트인 현재 네이버부스트캠프에서 진행 중인 프로젝트 Moyo에 대해 설명드리도록 하겠습니다.

[프로젝트 1] 실시간 화상회의 웹서비스 Moyo

저는 네이버 부스트캠프 웹·모바일 10기 과정에서 세 명의 팀원과 함께 6주간 Moyo(모여)라는 메타버스 기반 실시간 화상회의 서비스를 개발하고 있습니다. “모여”라는 이름처럼, 온라인 환경에서도 사람들이 자연스럽게 한 공간에 모여 소통하고 협업할 수 있는 가상 환경을 제공하는 서비스입니다.

부스트캠프를 진행하며 온라인 협업 환경에서 두 가지 근본적인 문제점을 직접 경험했습니다. 첫 번째는 자연스러운 관계 형성의 어려움이었습니다. 온라인에서는 오프라인처럼 복도에서 우연히 마주치거나, 점심시간에 자연스럽게 대화를 나누는 것이 불가능했습니다. 화상회의 도구에서는 대화 타이밍을 잡기 어렵고, 어색한 침묵이 발생하면 불안감이 커졌습니다. 팀 기반 학습 환경에서 팀원 간 라포 형성은 프로젝트 성공의 핵심인데, 기존 도구들은 이런 자연스러운 상호작용을 전혀 지원하지 못했습니다.

두 번째 문제는 협업 도구의 분산이었습니다. 조사 결과에 따르면 일반 직원은 평균 8.8개, 이사급은 10개 이상의 업무 툴을 사용하며, 탭 전환 후 재집중까지 약 23분이 소요된다고 합니다. 화상회의는 Zoom, 문서는 Notion, 코드 리뷰는 GitHub, 화이트보드는 Miro처럼 기능별로 분산된 도구들은 잦은 앱 전환을 유발하고 생산성 손실을 초래합니다. 저희는 이 두 문제를 해결하기 위해 Zep이나 Gather처럼 메타버스 환경에서 부스트캠프 생활 전체를 진행할 수 있는 통합 플랫폼을 기획했습니다.

실제 부스트캠프 커리큘럼을 토대로, 마스터님의 강의를 들을 수 있는 세미나실, 팀별 회의를 위한  회의실, 모여서 각자 코딩할 수 있는 모각코방, 개인 업무를 볼 수 있는 데스크 존, 그리고 우연한 만남을 유도하는 로비와 식당을 구현했습니다. 기술적으로는 Tiled 에디터로 맵을 제작하고, JavaScript 기반 2D 게임 엔진인 Phaser로 아바타 이동과 충돌 감지, 근접 사용자 자동 그룹화를 구현했습니다.

전체 아키텍처는 pnpm 기반 모노레포 구조로 설계했습니다. 프론트엔드는 React 19와 Vite 7을 사용하고, 상태 관리는 Zustand, 메타버스 렌더링은 Phaser 3, 화상통화는 LiveKit, 실시간 협업 코드 에디터는 Monaco Editor와 Yjs, 화이트보드는 Tldraw를 활용했습니다. 백엔드는 NestJS 11과 TypeORM, MySQL을 사용하고 Socket.io로 실시간 양방향 통신을 구현했습니다. 인프라는 Docker와 Docker Compose로 컨테이너화하고, GitHub Actions로 CI/CD 파이프라인을 구축했습니다. 모노레포를 선택한 이유는 프론트엔드와 백엔드 간 타입 정의를 공유하여 API 계약의 일관성을 보장하고, 단일 저장소에서 전체 시스템의 변경 이력을 추적할 수 있기 때문입니다.

프로젝트를 진행하며 여러 기술적 도전에 직면했고, 팀원들과 깊이 있는 논의를 통해 해결책을 찾아갔습니다.

첫 번째로 고민했던 것은 Phaser 게임 씬 내부 상태를 외부에 어떻게 안전하게 노출할 것인가였습니다. GameScene 내부에서 맵 관련 설정과 런타임 상태를 하나의 객체로 관리하고 있었는데, React UI 레이어에서 현재 맵 정보나 줌 단계를 읽어야 했습니다. 문제는 외부에서 실수로 이 값을 수정하는 것을 방지해야 한다는 점이었습니다. 네 가지 옵션을 검토했습니다. Getter로 깊은 복사본을 반환하는 방식은 원본 보호가 확실하지만 매번 복사본을 생성해야 해서 게임 루프에서 빈번하게 호출될 경우 GC 압박이 우려되었습니다. TypeScript의 Readonly 타입으로 노출하는 방식은 컴파일 단계에서 수정 시도를 잡아낼 수 있고 성능 저하가 없지만, 런타임에서는 여전히 변경 가능했습니다. 필드를 분리하는 방식은 외부 수정을 완전히 차단할 수 있지만 관련 정보가 흩어져 관리가 번거로웠습니다. 외부 접근을 완전히 차단하면 UI 연동 자체가 어려워졌습니다.

팀원들과 논의 끝에 Readonly 타입으로 노출하는 방식을 선택했습니다. 핵심 근거는 세 가지였습니다. 첫째, Phaser는 React와 달리 불변성보다 직접 조작을 권장하는 게임 엔진이므로 런타임 불변성을 위한 추가 오버헤드는 엔진의 설계 철학과 맞지 않았습니다. 둘째, 실수로 인한 수정은 대부분 개발 단계에서 발생하므로 TypeScript 컴파일 타임 보호로 충분했습니다. 셋째, 게임 루프는 60fps로 동작하며 매 프레임마다 상태를 조회할 수 있는데, 깊은 복사로 인한 성능 저하는 사용자 경험에 직접 영향을 미칩니다. 이 결정으로 외부에서의 실수를 방지하면서도 성능을 유지하는 균형점을 찾았습니다.

두 번째 도전은 서버 운영 방식 결정이었습니다. 무료 요금제인 micro 서버를 사용하면서 운영 서버 사양이 1 vCPU와 1GB RAM으로 제한되어 있었고, 이 환경에서 서버가 직접 빌드를 수행할 경우 메모리 부족으로 인한 장애 위험이 존재했습니다. 이에 따라 서버에서 애플리케이션을 어떤 방식으로 실행할지에 대한 선택이 필요했습니다.

운영 방식으로는 PM2를 사용해 Node.js 프로세스를 직접 실행하는 방식과 Docker 컨테이너 기반으로 서비스를 운영하는 방식을 비교했습니다. PM2 방식은 Docker 데몬으로 인한 추가 오버헤드가 없어 자원 사용 측면에서는 유리하지만, 서버마다 실행 환경을 직접 맞춰야 해 환경 일관성 확보가 어렵고 향후 서버 이전이나 확장 시 이식성이 떨어진다는 한계가 있었습니다. 반면 Docker 방식은 컨테이너 오버헤드와 이미지 관리 비용이 발생하지만, 실행 환경을 코드로 고정할 수 있어 배포 재현성과 롤백, 장기적인 운영 안정성 측면에서 장점이 있다고 판단했습니다.

최종적으로는 역방향 전환이 용이하다는 점을 핵심 판단 기준으로 삼아 Docker를 먼저 적용하기로 결정했습니다. Docker 환경에서 문제가 발생하더라도 PM2 방식으로 빠르게 전환할 수 있지만, 반대로 PM2를 선택한 뒤 Docker로 이전하려면 설정과 배포 구조를 다시 설계해야 했기 때문입니다. 또한 1 vCPU와 1GB RAM 환경에서 Docker 오버헤드가 실제로 문제가 될지는 이론만으로 판단하기 어렵다고 보았고, 제한된 환경에서도 충분히 검증 가능한 선택이라고 판단했습니다.

실제로 Docker를 적용한 결과, 저사양 서버에서도 서비스는 안정적으로 동작했고 배포 과정 역시 이미지 단위로 단순화할 수 있었습니다.

세 번째 도전은 근접 사용자 자동 그룹화에서 발생한 문제였습니다. Moyo의 핵심 기능 중 하나는 사용자들이 맵에서 인접한 위치로 이동하면 자동으로 화상통화 그룹이 형성되는 것입니다. 그런데 사용자가 경계선 근처에서 미세하게 움직일 때 그룹 가입과 탈퇴가 빠르게 반복되는 현상이 발생했습니다. 이로 인해 화상통화 연결이 계속 재연결되어 UX가 저하되고, SFU 서버에 지속적인 토큰 요청으로 과도한 요금 청구 위험이 있었으며, 불필요한 소켓 이벤트로 서버 부하가 증가했습니다.

이 문제를 해결하기 위해 하이스테리시스 로직을 바운더리 서비스에 구현했습니다. 하이스테리시스는 시스템이 이전 상태를 기억하여 작은 변화에 민감하게 반응하지 않도록 하는 기법으로, 온도 조절기의 원리와 동일합니다. 그룹 탈퇴 시 즉시 처리하지 않고 300밀리초 대기 후 최종 결정을 내리도록 했습니다. 사용자가 경계를 벗어나면 타이머가 시작되고, 300밀리초 이내에 다시 경계 안으로 돌아오면 탈퇴가 취소됩니다. 추가로 비활성 그룹 자동 정리 로직을 구현하여 메모리 누수도 방지했습니다. 결과적으로 동일한 시나리오 기반 측정에서 불필요한 API 요청이 90퍼센트 이상 감소했고, 화상통화 재연결 감소로 UX가 대폭 개선되었으며, SFU 서버 토큰 사용량도 절감되었습니다.

네 번째 도전은 유령 아바타 문제였습니다. Zustand를 사용한 상태 관리에서 본인 정보와 전체 사용자 목록을 별도의 상태로 관리하고 있었습니다. 서버로부터 사용자 목록을 수신한 후 두 상태를 순차적으로 업데이트하는 과정에서 타이밍 불일치가 발생했습니다. 사용자 A가 퇴장하고 서버에서 업데이트된 목록을 전송하면, 클라이언트에서 본인 정보를 먼저 업데이트하고 React 리렌더링이 발생하는데, 이 시점에 전체 사용자 목록에는 아직 퇴장한 사용자가 남아 있어서 화면에 유령 아바타가 잔존하는 현상이 발생했습니다. 이는 두 상태 간 일관성이 깨지는 경쟁 상태였습니다.

이 문제를 해결하기 위해 상태 업데이트를 원자적으로 처리하도록 구조를 재설계했습니다. 순차적으로 두 번 호출하던 것을 단일 set 함수 호출로 변경하여 본인 정보와 전체 사용자 목록을 동시에 업데이트했습니다. Zustand의 set 함수는 내부적으로 배치 처리되므로 단일 호출 내에서 여러 상태를 업데이트하면 하나의 리렌더링만 발생합니다. 결과적으로 테스트와 운영 환경에서 유령 아바타 이슈가 완전히 해결되었고, Chrome DevTools 프로파일링 결과 스프라이트 생성과 파괴 횟수가 감소하고 GC 발생 빈도도 줄어들었습니다.

다섯 번째 도전은 번들 최적화였습니다. Moyo는 Phaser, LiveKit, Monaco Editor, Tldraw 등 여러 대형 라이브러리를 사용합니다. 이 모든 라이브러리가 초기 번들에 포함되어 사이트 접속 시 전송 기준 4.5MB의 초기 로딩 부하가 발생했고, 약 1.2초의 FCP 지연이 발생하여 사용자 이탈 위험이 높아졌습니다.

세 가지 전략을 병행하여 번들을 최적화했습니다. 먼저 Vite 설정에서 대형 라이브러리별로 청크를 분리하여 필수 코드만 프리로드하고 대용량 청크는 선택적으로 제외했습니다. 다음으로 전역 컴포넌트에서 대형 라이브러리 참조를 타입 수준으로 분리하여 의존성 오염과 번들 비대화를 방지했습니다. 마지막으로 무거운 Context Provider를 페이지 단위 지연 로딩 컴포넌트 내부로 이동시켜 메인 번들 의존성을 원천 차단했습니다. 결과적으로 초기 로딩 전송량이 약 90퍼센트 절감되어 4.5MB에서 512KB 미만으로 줄었고, 약 1,200밀리초의 로딩 지연이 제거되었습니다.

여섯 번째 도전은 CI/CD 파이프라인 구축이었습니다. 기존 배포 방식은 GitHub Actions에서 서버로 SSH 접속하여 명령을 실행하는 구조였는데, GitHub Secrets에 서버 SSH Private Key를 저장해야 하는 보안 리스크가 있었고, 저사양 서버에서 직접 빌드 시 메모리 부족 위험도 있었습니다.

그래서 저는 빌드와 배포를 분리하는 아키텍처를 설계했습니다. 빌드는 GitHub에서 제공하는 러너에서 수행하고, 운영 서버에는 Self-hosted Runner를 설치하여 배포만 담당하도록 했습니다. 이렇게 하면 GitHub에서 서버로 SSH 접속하는 대신 서버가 GitHub로 아웃바운드 연결을 맺는 구조가 되어, 서버 SSH Key를 GitHub에 노출하지 않고 인바운드 포트도 불필요해집니다. 결과적으로 운영 서버에서 빌드 작업이 완전히 제거되어 서버 가용성이 향상되었고, 캐시를 활용하여 전체 파이프라인 시간이 8에서 12분에서 3에서 4분으로 단축되었으며, 보안 리스크도 대폭 감소했습니다.

이 프로젝트를 통해 실시간 협업 시스템의 복잡성을 직접 경험했습니다. 단순히 기능을 구현하는 것을 넘어, 왜 이 기술을 선택했는지, 어떤 트레이드오프가 있는지를 팀원들과 논의하며 결정하는 과정이 가장 값진 배움이었습니다. WebSocket 기반 실시간 통신 아키텍처 설계, CRDT를 활용한 협업 편집 구현, WebRTC와 SFU 기반 화상통화 시스템 통합, Docker 컨테이너화와 CI/CD 파이프라인 구축, 프론트엔드 번들 최적화와 성능 튜닝 등 다양한 영역에서 깊이 있는 경험을 쌓았습니다.

그리고 팀원들과 6주간 긴밀하게 협업하며 기술적 의사결정의 중요성도 배웠습니다. 각자의 의견을 GitHub Discussions에 문서화하고, 장단점을 비교 분석한 뒤 합의에 도달하는 과정을 통해 근거 기반 의사결정의 가치를 체감했습니다. Moyo를 통해 경험한 실시간 시스템 설계 역량을 바탕으로, 앞으로 더 복잡한 분산 시스템과 대규모 트래픽 처리에 도전하고 싶습니다.

**[프로젝트 2]** 주점 프로젝트

2025년 2학기, 대학교 주점을 위한 주문 서비스를 운영한 경험이 있습니다. BE를 맡았고 프레임워크는 Express를 사용하였습니다. Express는 Spring에 비해 가볍고 빠른 개발이 가능하다는 장점이 있습니다. 결제와 같은 서비스는 사업자 등록이 필요하기도 했기에 보안 및 완전한 트랜잭션 관리에 강점이 있는 Spring을 사용하기보다는 유연한 개발을 위해 Express를 사용했습니다. 서버 부분을 혼자 개발하게 되면서 Docker를 사용하기보다는 리소스를 덜 잡아먹고 빠르게 실행 가능한 PM2를 사용하는 선택을 했습니다. 가장 신경 썼던 부분은 주문을 하게 되는 과정에서 손님의 수고스러움을 덜어내려는 과정이었습니다. 일반적으로 요즘은 매장에 주문 키오스크가 달려 있어 쉽고 빠르게 주문이 가능합니다. 하지만 주점 특성상, 키오스크라는 하드웨어를 구비하기 힘들었고 메뉴판을 직접 서빙하여 주문을 받고 주방에 전달하는 일련의 과정이 매우 불편하게 느껴졌고 이 서비스의 필요성을 느끼게 되었습니다.

첫 번째로 고민했던 것은 QR 코드 기반 세션 관리 시스템이었습니다. 손님이 테이블에 부착된 QR 코드를 스캔하면 해당 테이블에 대한 주문 세션이 자동으로 생성됩니다. 이때 발생할 수 있는 문제는 이전 손님의 세션 토큰이 아직 유효한 상태에서 새 손님이 같은 테이블에 앉는 경우입니다. 이를 해결하기 위해 테이블마다 현재 활성 세션을 단 하나만 유지하는 구조를 설계했습니다. 새 세션이 열리면 기존 세션은 자동으로 만료 처리되고, 미들웨어에서 세션 토큰이 현재 테이블의 활성 세션인지 검증합니다. 이렇게 하면 이전 손님의 토큰으로는 더 이상 주문할 수 없어 테이블 간 주문 혼선을 방지할 수 있습니다.

두 번째 개선은 포장 주문을 위한 멀티 세션 지원이었습니다. 매장 주문은 테이블당 하나의 세션만 허용하는 것이 맞지만, 포장 주문의 경우 여러 손님이 동시에 주문할 수 있어야 합니다. 이를 위해 포장 전용 가상 테이블을 만들고, 이 테이블에 대해서만 복수 세션을 허용하는 로직을 분리했습니다. 포장 세션은 일반 세션보다 짧은 TTL을 적용하여 불필요한 세션이 오래 남아있지 않도록 했습니다. 또한 포장 주문에는 10퍼센트 할인을 자동 적용하여 포장을 유도하는 정책도 시스템에 반영했습니다.

세 번째 개선은 주문 상태를 실시간으로 주방에 전달하는 SSE 기반 알림 시스템이었습니다. 손님이 주문을 생성하면 주방 관리 화면에 즉시 새 주문이 표시되어야 합니다. 이를 위해 Server-Sent Events를 활용한 단방향 실시간 통신을 구현했습니다. 관리자 클라이언트가 SSE 엔드포인트에 연결하면 서버는 해당 연결을 유지하다가 주문 생성, 상태 변경 등의 이벤트가 발생할 때마다 모든 연결된 클라이언트에게 브로드캐스트합니다. 25초 간격의 하트비트를 통해 연결 유지 상태를 확인하고, 끊긴 클라이언트는 자동으로 정리됩니다. WebSocket 대신 SSE를 선택한 이유는 주문 알림이 서버에서 클라이언트로의 단방향 통신이므로 SSE의 단순함이 더 적합했기 때문입니다.

네 번째 개선은 주문 상태를 명확한 상태 머신으로 관리한 것이었습니다. 주문은 PENDING에서 시작하여 CONFIRMED, IN_PROGRESS, SERVED 순서로 진행되거나 중간에 CANCELED로 전환될 수 있습니다. 각 상태에서 허용되는 전이만 가능하도록 상태 머신을 정의하여 잘못된 상태 변경을 원천 차단했습니다. 예를 들어 이미 서빙된 주문을 다시 조리 시작 상태로 변경하거나, 취소된 주문을 다시 진행하는 것은 불가능합니다. 또한 주문 확정 시에는 재고를 차감하고, 확정된 주문을 취소할 때는 재고를 복구하는 트랜잭션 로직도 상태 전이와 함께 원자적으로 처리했습니다.

다섯 번째 개선은 주방 대시보드에서 긴급 주문을 한눈에 파악할 수 있도록 한 것이었습니다. 활성 주문 조회 API에서는 주문을 세 가지 카테고리로 분류하여 반환합니다. 15분 이상 경과한 주문은 긴급으로, 확정되었지만 아직 조리 시작 전인 주문은 대기로, 조리 중인 주문은 조리 중으로 분류됩니다. 각 카테고리 내에서는 오래된 주문이 먼저 표시되도록 정렬하여 주방에서 우선순위를 쉽게 파악할 수 있습니다. 주문 생성 시점을 기준으로 경과 시간을 계산하여 응답에 포함시켜 프론트엔드에서 별도 계산 없이 바로 표시할 수 있도록 했습니다.

여섯 번째 개선은 Notion 데이터베이스와의 연동이었습니다. 주점 운영진이 실시간 매출 현황을 Notion에서 확인하고 싶다는 요구사항이 있었습니다. 주문이 생성되면 백그라운드에서 Notion API를 호출하여 해당 주문 정보를 Notion 데이터베이스에 기록합니다. 이때 세트 메뉴의 경우 개별 품목으로 펼쳐서 기록하여 실제 조리해야 할 품목 수를 정확히 파악할 수 있도록 했습니다. Notion 동기화는 트랜잭션 외부에서 비동기로 수행하여 Notion API 장애가 주문 생성에 영향을 주지 않도록 설계했습니다. 메인 주문 처리의 안정성을 보장하면서도 부가 기능을 제공하는 방식입니다.

일곱 번째 개선은 동시성 문제에 대한 대응이었습니다. 주점 특성상 특정 시간대에 주문이 몰릴 수 있고, 같은 테이블에서 여러 사람이 동시에 주문을 시도할 수 있습니다. 재고 차감, 세션 상태 변경 등의 작업에서 데드락이나 락 타임아웃이 발생할 수 있습니다. 이를 위해 지수 백오프와 지터를 적용한 재시도 로직을 구현했습니다. 데드락이나 락 타임아웃 에러가 감지되면 최대 4회까지 재시도하되, 재시도 간격은 지수적으로 증가하고 무작위 지터를 추가하여 동시 재시도로 인한 충돌을 줄였습니다.

이 프로젝트를 통해 실제 운영 환경에서 사용자 경험을 개선하기 위한 다양한 고민을 할 수 있었습니다. 단순히 기능을 구현하는 것을 넘어, 손님이 QR 코드를 스캔하고 주문을 완료하기까지의 전 과정에서 발생할 수 있는 불편함과 오류 상황을 예측하고 대응하는 것이 중요했습니다. 또한 주방 운영진의 입장에서 주문을 효율적으로 처리할 수 있도록 정보를 구조화하고 우선순위를 명확히 하는 것도 사용성 개선의 핵심이었습니다. 하드웨어 투자 없이 손님 개인의 스마트폰만으로 주문이 가능한 시스템을 구축함으로써, 임시로 운영되는 주점 환경에서도 효율적인 주문 처리가 가능하다는 것을 증명할 수 있었습니다
